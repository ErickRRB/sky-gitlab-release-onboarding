Que skill usar para GitLab, Stage y tags

Esta pagina resume los casos comunes del flujo Sky. La idea es que alguien nuevo pueda mirar el problema, elegir la skill correcta y evitar los errores caros: tag mal apuntado, branch sucia, MR con squash incorrecto o release construido desde el lugar equivocado.

Decision rapida

Arranca por el caso de uso, no por el nombre de la rama.

branch limpia

Un ticket esta mezclado en main

Usar /feature-branch. Extrae el diff real desde un tag base, separa codigo y bump, crea el tag antes del merge.

commit / MR

Ya tengo cambios listos

Usar /sky-commit-changes y despues /sky-create-mr. En V2 usar las variantes /sky-v2-*.

deploy

Dejar un repo listo para QA/Stage

Usar /sky-prepare-deploy. Calcula version, changelog, lockfile, validacion, commit, push y Step Jira.

epica

Release de varios tickets/repos

Usar /sky-generate-release-notes y luego /sky-create-release-branch.

retag

Hay que mover tags existentes

Usar /sky-rebuild-tag-chain. No es un branch normal: preserva ancestry entre tags y crea backups.

repos

Falta ubicar o clonar un repo

Usar /sky-gitlab-search-repos o /sky-gitlab-clone-repo.

Flujo Stage normal

Sirve cuando hay que aislar un ticket o dejar una rama de deploy para Stage sin arrastrar commits ajenos.

1

/feature-branch VC-XXXX

Detecta repo, tag base, commits candidatos y si el caso en realidad debe ir a tag-chain. No usa cherry-pick: aplica el diff neto del cambio.

2

Commit de codigo y commit de release

El codigo queda separado del bump. El commit de bump contiene changelog, version y lockfile si corresponde.

3

/sky-tag-creator

Crea el tag anotado vX.Y.Z sobre el commit de release. El mensaje sale del changelog de ese commit.

4

Merge origin/main con --no-ff

Se hace despues del tag. El MR final no debe usar squash, porque la historia codigo -> bump/tag -> merge tiene significado.

Si el pedido menciona recrear tags, retag, tag mayor, mismo contenido con otra version o hotfix sobre tag anterior, no sigas con este flujo: cambia a /sky-rebuild-tag-chain.

Release por epica o multi-ticket

Este camino arma primero la evidencia y despues crea ramas de release por repo.

documentar

/sky-generate-release-notes

Lee tickets, identifica repos, busca MRs abiertos o mergeados, lista commits por repo y genera release_notes.md. Si hay MRs pendientes, los marca como pendientes.

branch release

/sky-create-release-branch

Toma el release_notes.md, crea release/vX.Y.Z, aplica los commits, actualiza changelog/version y publica la rama. Para tags delega en /sky-tag-creator.

Rearmar tags ya creados

Usar cuando hay que insertar un hotfix en un tag anterior y hacer que tags posteriores queden construidos encima de ese tag corregido.

1

Confirmar el estado final esperado

Antes de tocar refs, la skill pide confirmar base tag, tags a recrear en orden, source refs/backups y estrategia de changelog.

2

Snapshot y backups

Verifica refs locales/remotas usando commits peeled vX.Y.Z^{} y crea tags backup/... antes de mover nada.

3

Reconstruir cada tag en orden

Para cada tag crea delta historico, bump/changelog y tag anotado. Valida que el tag anterior sea ancestro del siguiente.

4

Push solo con aprobacion

Tags usan --force solo despues de validacion y aprobacion. Branches usan --force-with-lease. El MR va sin squash.

El objetivo no es solo que los archivos queden iguales. El objetivo es que la historia Git quede correcta: base corregida -> tag 1 -> tag 2 -> HEAD.

Release-Unit V2

Usar cuando main funciona como QA y Stage/Prod debe promover solo una parte de lo que ya entro a main.

metadata

/sky-v2-release-unit

Resuelve Release-Unit, label Jira ru-*, epica, scope y base tag.

implementar

/sky-v2-take-ticket

Implementa ticket con gate de Release-Unit. Si la metadata esta pendiente, la deja visible sin bloquear codigo cuando el scope tecnico es claro.

commit

/sky-v2-commit-changes

Crea commit con trailers parseables: Release-Unit, Epic, Refs, Deploy-Scope y flags.

MR

/sky-v2-create-mr

Abre MR con metadata. Feature/fix normal debe squashar a un commit parseable; release/tag-chain no debe usar squash.

release

/sky-v2-prepare-release

Genera release branch desde baseTag y manifest, cherry-pickeando solo commits de la Release Unit.

soporte tags

/sky-tag-creator

Sigue siendo obligatorio para tags reales. V2 no crea tags a mano con git tag -m.

Casos edge que las validaciones evitan

Estos son errores frecuentes cuando el flujo se hace manualmente. Las skills no solo ejecutan comandos: tambien ponen checks para detectar el problema antes de publicar branch, tag o MR.

base incorrecta

Usar un tag local viejo como base

Que pasa manualmente
La rama se arma sobre un tag local que ya no coincide con remoto.
Check que lo evita
git fetch --tags, refs remotas y commit peeled vX.Y.Z^{}.
Skill
/feature-branch, /sky-rebuild-tag-chain, /sky-v2-prepare-release.
tag mal apuntado

Crear el tag sobre el merge commit

Que pasa manualmente
El tag incluye cambios de main que no pertenecen al release.
Check que lo evita
create-tag.sh, dod-check.sh y validacion de peeled commit.
Skill
/sky-tag-creator dentro del flujo padre.
mensaje tag

Reusar un mensaje de tag viejo

Que pasa manualmente
El tag queda con notas stale, bullets no linkeados o texto que no existe en changelog.
Check que lo evita
Mensaje generado desde la seccion [X.Y.Z] del changelog en el release commit.
Skill
/sky-tag-creator.
formato tag

Publicar STAGE-X.Y.Z o sufijos raros

Que pasa manualmente
CI, changelog y deploy links no matchean el contrato Sky.
Check que lo evita
Validacion estricta de formato vX.Y.Z.
Skill
/sky-tag-creator, /sky-prepare-deploy.
changelog

Duplicar o borrar versiones historicas

Que pasa manualmente
Al resolver merge, se pisa el changelog de main o se duplica [X.Y.Z].
Check que lo evita
git diff origin/main..HEAD -- changelog.md y regla de preservar historia de main.
Skill
/feature-branch, /sky-rebuild-tag-chain.
drag-along

Cherry-pick arrastra commits ajenos

Que pasa manualmente
Un commit que venia con merges trae archivos de otros tickets.
Check que lo evita
Extraccion de diff neto y revision de archivos por commit/autor.
Skill
/feature-branch.
staging

git add . sube ruido o secretos

Que pasa manualmente
Entran .env, logs, dumps, notas locales o cambios de otro ticket.
Check que lo evita
Staging por archivo, diff staged revisado y exclusion explicita de ruido.
Skill
/sky-commit-changes, /sky-v2-commit-changes.
runtime

Validar con Node incorrecto

Que pasa manualmente
Build o lockfile pasan/fallan por version local, no por la version real del repo.
Check que lo evita
Detectar .nvmrc, .node-version o engines.node antes de validar.
Skill
/feature-branch, /sky-prepare-deploy, /sky-v2-take-ticket.
lockfile

Omitir lockfile cuando cambio dependencias

Que pasa manualmente
El tag publica package.json nuevo pero CI falla con lockfile desalineado.
Check que lo evita
npm ci sobre worktree del tag y lockfile-regen.sh.
Skill
/feature-branch, /sky-prepare-deploy.
retag

Force-push de tag sin backup

Que pasa manualmente
Se pierde la referencia anterior y no queda forma simple de auditar o volver atras.
Check que lo evita
Backup backup/vX.Y.Z-before-... y verificacion object/peeled antes del push.
Skill
/sky-rebuild-tag-chain, /sky-tag-creator.
ancestry

Tags correctos en archivos pero no en historia

Que pasa manualmente
El contenido parece bien, pero un tag posterior no desciende del hotfix base.
Check que lo evita
verify-tag-chain.sh y git merge-base --is-ancestor.
Skill
/sky-rebuild-tag-chain.
squash

Squash en MR de release/tag-chain

Que pasa manualmente
Se destruye la secuencia feature -> bump/tag -> merge que justifica el release.
Check que lo evita
Clasificacion de branch y regla de squash disabled para release/tag-chain.
Skill
/sky-create-mr, /sky-v2-create-mr.
ambiguedad

Renombrar tag mayor sin definir changelog

Que pasa manualmente
Se mezcla metadata-only bump con una promocion semantica de release.
Check que lo evita
Outcome gate: tipo, source tag, new tag, categorias, excepciones y MR esperado.
Skill
/sky-rebuild-tag-chain.
V2

Taggear todo main cuando main es QA

Que pasa manualmente
Stage/Prod recibe commits aprobados para QA pero no pertenecientes al deploy.
Check que lo evita
Manifest desde baseTag + trailers Release-Unit.
Skill
/sky-v2-release-unit, /sky-v2-prepare-release.
trailers

Squash commit sin metadata V2

Que pasa manualmente
Despues no se puede generar manifest confiable para release parcial.
Check que lo evita
git interpret-trailers --parse y auditoria release-unit-audit.py.
Skill
/sky-v2-commit-changes, /sky-v2-create-mr.
conflictos

Dejar conflict markers en el merge

Que pasa manualmente
El branch compila o publica con <<<<<<< en archivos fuente o JSON.
Check que lo evita
Busqueda de markers <<<<<<<|=======|>>>>>>> antes del push final.
Skill
/feature-branch, /sky-rebuild-tag-chain.

Reglas criticas

No push a main/master

Siempre trabajar en branch. Si hay que publicar, se pushea la branch o tag, nunca main directo.

Tags antes del merge main

El tag apunta al commit de bump/release. El merge --no-ff origin/main viene despues.

Peeled commit siempre

Para validar tags anotados usar git rev-parse vX.Y.Z^{}, no solo el hash del objeto tag.

No squash en release/tag-chain

La ancestry importa. Squash destruye la estructura que permite auditar tags y merges.

Backups antes de retag

Antes de mover un tag remoto, crear y validar backup/vX.Y.Z-before-....

Preguntar si el outcome es ambiguo

En tag mayor, mismo contenido u otra version, confirmar changelog, source tag, target tag y MR esperado antes de tocar refs.
